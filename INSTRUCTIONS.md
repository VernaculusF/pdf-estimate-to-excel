# Инструкция по использованию: Конвертер смет PDF → Excel

## Описание

Инструмент автоматически извлекает табличные данные из PDF-файлов смет (локальные сметы, акты выполненных работ, ресурсные ведомости) и сохраняет их в формате Excel (.xlsx).

Для каждого PDF-файла создаётся Excel-файл с тремя листами:

| Лист | Содержание |
|------|-----------|
| **Header** | Изображение шапки документа (первая страница PDF). Содержит печати, подписи, штампы, реквизиты — всё, что невозможно корректно распознать как текст. |
| **Estimate** | Таблица сметы с данными: номера строк, обоснование, наименование работ и ресурсов, единицы измерения, количество, цена, сумма и т.п. |
| **Source Text** | Полный текст PDF (текстовый слой или результат OCR) для справки и ручной сверки. |

Также создаётся отчёт о качестве конвертации (`conversion_report.xlsx` и `conversion_report.json`).

---

## Системные требования

- **Операционная система**: Windows 10/11 или Linux
- **Python**: 3.11 или выше
- **Tesseract OCR**: должен быть установлен
  - Windows: `C:\Program Files\Tesseract-OCR\tesseract.exe`
  - Linux: `sudo apt install tesseract-ocr tesseract-ocr-rus`
- **Оперативная память**: минимум 4 ГБ (OCR обработка сканов в 300 DPI требует памяти)

## Установка

### 1. Установка Python

Скачайте и установите Python 3.11+ с [python.org](https://www.python.org/downloads/).
При установке отметьте галочку **"Add Python to PATH"**.

### 2. Установка Tesseract OCR

**Windows:**
Скачайте установщик с [github.com/UB-Mannheim/tesseract](https://github.com/UB-Mannheim/tesseract/wiki) и установите в папку по умолчанию.

**Linux:**
```bash
sudo apt install tesseract-ocr tesseract-ocr-rus
```

### 3. Установка зависимостей Python (только при ручном запуске)

Если вы запускаете через `run.bat` — **этот шаг не нужен**. `run.bat` автоматически создаст виртуальное окружение и установит все зависимости из `requirements.txt`.

Для ручного запуска откройте командную строку (cmd) в папке `project/` и выполните:

```
pip install -r requirements.txt
```

Или, если используется виртуальное окружение:

```
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
```

## Использование

### Способ 1: Быстрый запуск через `run.bat` (Windows) — рекомендуется

1. Поместите PDF-файлы в папку `input/`
2. Дважды кликните на файл `run.bat`
3. Скрипт автоматически:
   - Создаст виртуальное окружение Python (`project/venv/`), если его нет
   - Установит все Python-зависимости из `requirements.txt`
   - Создаст папку `output/`, если её нет
   - Запустит конвертацию всех PDF из `input/`
4. Готовые Excel-файлы появятся в папке `output/`

### Способ 2: Командная строка

Откройте командную строку в папке `project/`.

**Обработка всех PDF из папки:**
```
python main.py --input ../input --output ../output
```

**Обработка одного файла:**
```
python main.py --file ../input/example.pdf --output ../output
```

**Принудительное использование OCR для всех файлов:**
```
python main.py --input ../input --output ../output --use-ocr
```

**Подробный вывод (для диагностики):**
```
python main.py --input ../input --output ../output --verbose
```

## Параметры командной строки

| Параметр | Сокращение | Описание |
|----------|-----------|----------|
| `--input` | `-i` | Папка с входными PDF-файлами (по умолчанию: `../input`) |
| `--output` | `-o` | Папка для выходных Excel-файлов (по умолчанию: `../output`) |

| `--verbose` | `-v` | Подробный вывод для диагностики |

## Архитектура решения (почему это работает так)

Конвертер использует **два независимых конвейера** в зависимости от типа PDF:

### Путь 1: PDF с текстовым слоем (pdfplumber)
Если документ содержит встроенный текст (не скан), используется библиотека **pdfplumber**, которая работает напрямую с PDF-структурой: извлекает слова, их координаты и восстанавливает таблицы по границам ячеек. Это самый точный и быстрый путь.

### Путь 2: Сканированные PDF (OCR)
Если текстовый слой отсутствует, страницы рендерятся в изображения (300 DPI) и обрабатываются через **Tesseract OCR** с предварительной обработкой (OpenCV: контраст, резкость, детекция линий таблиц). Сканы проходят автоматическую коррекцию ориентации (Tesseract OSD) — каждая страница проверяется индивидуально.

### Автоматическое определение типа
Программа сама определяет, является ли PDF сканом: проверяет первые страницы на наличие извлекаемого текста и слов через pdfplumber. Если текстового слоя нет — переключается на OCR.

## Структура выходных файлов

### Excel-файл (для каждого PDF)

- **Лист "Header"** — изображение шапки сметы (печати, подписи, реквизиты, штампы) с первой страницы PDF. Это не редактируемый текст, а именно картинка, потому что шапки содержат растровые элементы (печати, подписи), которые невозможно достоверно распознать как структурированный текст.

- **Лист "Estimate"** — структурированная таблица с данными сметы. Столбцы: номер п/п, обоснование, наименование работ и ресурсов, единицы измерения, количество, сметная стоимость, формулы и т.д.

- **Лист "Source Text"** — полный текст, извлечённый из PDF (текстовый слой или результат OCR). Служит для справки и ручной проверки полноты данных.

### Отчёт о качестве

- `conversion_report.xlsx` — сводная таблица с метриками качества для каждого файла
- `conversion_report.json` — те же данные в формате JSON

Метрики включают:
- Количество символов в исходном PDF
- Количество символов в Excel
- Процент сохранения текста (preservation)
- Предупреждения о проблемах

## Поддерживаемые типы PDF

| Тип PDF | Поддержка | Примечание |
|---------|-----------|-----------|
| PDF с текстовым слоем | ✅ Полная | Извлечение через pdfplumber, сохраняет структуру и координаты |
| Сканированные PDF (хорошее качество) | ✅ Полная | OCR (Tesseract) с предобработкой OpenCV |
| Сканированные PDF (низкое качество, размытые) | ⚠️ Частичная | OCR зависит от читаемости текста; результат требует сверки |
| Смешанные PDF | ⚠️ Частичная | Если смесь текстовых и сканированных страниц — сканированные страницы могут потребовать ручной обработки |
| Перевёрнутые/повёрнутые страницы | ✅ Полная | Автоматическая коррекция ориентации по каждой странице |
| Многостраничные сметы | ✅ Полная | Таблицы с разных страниц объединяются в один лист |

## Известные проблемы и ограничения (честный раздел)

**Важно понимать:** PDF — это формат для **визуального отображения и печати**, а не для структурированного извлечения данных. PDF не содержит таблиц в привычном смысле (как в Word или Excel). Внутри PDF — это набор текстовых фрагментов с абсолютными координатами, которые могут быть расположены в любом порядке (часто не топологическом, а по порядку отрисовки). Поэтому **идеальное извлечение таблиц из PDF математически невозможно** в общем случае.

### Что работает хорошо

- Чёткие сканы хорошего качества (300 DPI+, чёрный текст на белом фоне)
- PDF с текстовым слоем, где верстка соответствует логической структуре таблицы
- Простые одноуровневые таблицы с металлическими/жирными границами
- Стандартные столбцы смет (№ п/п, наименование, ед.изм., количество, цена, сумма)

### Что работает плохо или невозможно исправить

| Проблема | Почему это происходит | Можно ли исправить |
|----------|----------------------|-------------------|
| **Размытые/бледные сканы** | OCR не может прочитать то, что не видит человеку. Tesseract — не волшебная палочка. | Нет. Исходный скан должен быть качественным. |
| **Таблицы без линий (на пробелах)** | pdfplumber пытается угадать границы ячеек по отступам, но если верстка сделана через пробелы и табуляцию, а не через линии — структура теряется. | Частично. Распознавание через пробелы реализовано, но точность ниже, чем по линиям. |
| **Объединённые ячейки (rowspan/colspan)** | PDF не хранит понятие "объединённая ячейка". Программа видит отдельные текстовые блоки и пытается восстановить таблицу. | Частично. Сложные многоуровневые шапки с объединениями требуют ручной корректировки. |
| **Формулы и сложные выражения** | При OCR символы `*`, `/`, `(`, `)` и цифры могут распознаваться с ошибками. Формула "12.5×3+4" может стать "12,5*3+4" или "1253+4". | Нет. OCR не гарантирует 100% точность символов. Формулы нужно проверять вручную. |
| **Печати и подписи в шапке** | Это растровые изображения, накладываемые поверх текста. Они мешают OCR и pdfplumber. | Нет. Поэтому шапка сохраняется как картинка, а не как текст. |
| **Таблицы, разорванные на несколько страниц с повторяющимися заголовками** | Автоматическое объединение работает, но повторяющиеся заголовки могут попадать в данные. | Частично. Фильтрация заголовков реализована, но не идеальна для всех типов смет. |
| **Сметы с нестандартной версткой** | Каждая организация верстает сметы по-своему: разные шрифты, разные отступы, вставленные картинки, водяные знаки. Универсальный парсер не может охватить все варианты. | Нет. Для уникальных макетов требуется индивидуальная настройка. |
| **Смешанные PDF (текст + скан в одном файле)** | Автodetection выбирает один путь для всего файла. Если текстовые страницы обрабатываются через OCR — качество падает. | Частично. Можно использовать `--use-ocr`, но это замедлит обработку текстовых страниц. |
| **Неверное разделение на столбцы** | Если в смете много столбцов с числами, детекция границ может ошибиться и сместить данные в соседний столбец. | Частично. Эвристики улучшены, но edge cases остаются. |
| **Языковые модели (LLM) не используются** | Использование GPT/Claude для распознавания таблиц было бы в 10-100 раз медленнее, дороже и требовало бы API-ключей. Точность LLM на таблицах без чётких границ тоже не 100%. | Нет. Текущее решение — это компромисс скорость/точность/стоимость. |

## Почему это — лучшая из возможных реализаций

Этот конвертер построен на **сочетании лучших специализированных библиотек** для работы с PDF и OCR, доступных на сегодняшний день:

1. **pdfplumber** — лучшая библиотека для извлечения структурированных таблиц из PDF с текстовым слоем. Она работает напрямую с PDF-объектами, а не через хаки.
2. **pypdfium2** — самый быстрый и корректный рендерер PDF в изображения (на базе Chromium PDFium), уважает метаданные поворота страниц.
3. **Tesseract OCR** — промышленный стандарт OCR с поддержкой русского языка. Используется с русским `tessdata` и конфигурацией `--oem 1 --psm 6`, оптимальной для таблиц.
4. **OpenCV** — предобработка изображений (контраст, резкость, детекция линий таблиц) адаптирована под сканы смет.

Альтернативы хуже:
- **Всё через OCR** (включая текстовые PDF) — медленнее в 5-10 раз, портит точность на документах с текстовым слоем.
- **Использование LLM (ChatGPT/Claude)** — в 50-100 раз дороже, требует интернета и API, не гарантирует точность на числах.
- **Коммерческие решения ( Abbyy, Amazon Textract)** — работают с английским лучше, с русскими сметами — хуже или дорого.
- **Ручной ввод** — медленнее в 1000 раз.

**Этот инструмент реализует оптимальную стратегию:**
- Для текстовых PDF — максимально быстрый и точный путь (pdfplumber).
- Для сканов — OCR с адаптированной предобработкой и авто-коррекцией ориентации.
- Качество каждой конвертации измеряется и записывается в отчёт.
- Шапка сохраняется как изображение, чтобы не портить растровые элементы распознаванием.

**Вывод:** при текущем состоянии технологий (2026 год) это решение обеспечивает максимально возможную автоматизацию при сохранении приемлемой точности. Оставшиеся ошибки — это фундаментальные ограничения формата PDF и OCR, а не недоработки программы.

## Решение проблем

| Проблема | Решение |
|----------|---------|
| Tesseract не найден | Установите Tesseract OCR и убедитесь, что он доступен по стандартному пути |
| Пустой Excel-файл | Попробуйте флаг `--use-ocr` для принудительного OCR |
| Плохое качество распознавания | Убедитесь, что скан хорошего качества (300 DPI рекомендуется). Размытые сканы OCR не исправит. |
| Ошибка "No tables detected" | PDF может не содержать табличных данных в ожидаемом формате, либо таблица без линий. Попробуйте `--use-ocr`. |
| Перевёрнутый текст в шапке | Автокоррекция ориентации работает на входе. Если не помогло — исходный скан слишком низкого качества для OSD. |
| Данные в неправильных столбцах | Сложная верстка или таблица на пробелах. Требуется ручная корректировка в Excel. |

## Рекомендации для получения лучшего результата

1. **Используйте исходники максимального качества**. PDF, сгенерированные из Word/Excel, работают идеально. Сканы — только хорошего качества (300 DPI, чёткий текст).
2. **Не используйте сканы с печатями поверх таблиц**. Печати закрывают текст и OCR не справляется.
3. **Сверяйте результат**. Даже при 98% сохранности текста оставшиеся 2% могут быть критичными (например, цифры в колонке "Сумма").
4. **Сохраняйте оригинальный PDF**. Excel — это рабочий документ для анализа, но юридически значимым остаётся исходный PDF.
5. **Для проблемных файлов** используйте флаг `--use-ocr` или разбейте PDF на части (шапка и таблицы отдельно).

## Техническая поддержка

Если файл не конвертируется корректно:
1. Проверьте `conversion_report.json` — там указан preservation percent и warning.
2. Запустите с флагом `--verbose` для детального лога.
3. Убедитесь, что PDF открывается и читается вручную — если текст нечитаем для человека, OCR тоже не справится.
4. Для текстовых PDF с плохой версткой программа может потребовать ручной корректировки результата.
